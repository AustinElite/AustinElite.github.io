可观测性不是多装几个监控面板，而是让团队能够从系统输出推断内部状态。对于现代后端系统，问题往往不会停留在单个服务里：一次慢请求可能穿过网关、业务服务、缓存、消息队列、数据库和第三方 API。没有完整的观测链路，排查就会变成猜谜。

好的可观测性应该回答三个问题：发生了什么、影响了谁、下一步去哪看。

## Logs

日志需要结构化字段、请求标识和明确的错误上下文，避免只记录一段无法聚合的自然语言。

推荐至少包含这些字段：

- `traceId`：串联一次请求或任务。
- `spanId`：标识当前调用片段。
- `userId` 或匿名主体 ID：用于定位影响范围。
- `service`、`env`、`version`：用于判断部署和环境。
- `errorCode`：比纯文本异常更适合统计。
- `durationMs`：方便和指标互相印证。

日志不是越多越好。高频正常日志会制造噪声，也会增加存储成本。关键路径、异常路径和状态变化更值得记录。

## Metrics

从延迟、流量、错误和饱和度开始，再加入业务指标。这四类基础指标可以快速判断系统健康度：

- **Latency**：请求耗时、P95、P99。
- **Traffic**：QPS、消息吞吐、任务数量。
- **Errors**：错误率、异常类型、失败原因。
- **Saturation**：CPU、内存、连接池、队列堆积。

业务指标同样重要。例如支付成功率、订单创建量、Agent 任务完成率、搜索召回为空的比例。技术指标告诉你系统是否运行，业务指标告诉你产品是否真的可用。

## Traces

跨服务链路追踪可以帮助判断延迟发生在哪一跳，但必须控制采样和敏感数据。

一次 Trace 应该能展示完整路径：网关接收请求、服务 A 查询缓存、服务 B 调用数据库、服务 C 调用第三方接口。排查慢请求时，Trace 能直接指出耗时集中在哪个 Span。

需要注意的是，Trace 不应该记录敏感请求体、密码、Token 或完整个人信息。采样策略也要分层：普通请求低采样，错误请求和慢请求高采样。

## 告警设计

告警的目标不是“响得越多越安全”，而是及时发现真正影响用户的问题。建议从 SLO 出发，例如：

- 99% 的 API 请求在 500ms 内返回。
- 支付链路 5 分钟错误率低于 0.5%。
- 消息队列积压不超过 10 分钟。
- Agent 任务完成率不低于 98%。

告警应该包含影响范围、当前值、阈值、可能原因和排查入口。只写“服务异常”的告警没有帮助，值班人员还得重新找上下文。

## Dashboard

仪表板需要围绕排查路径组织，而不是把所有图表堆在一起。一个实用的后端仪表板可以分成：

1. 总览：流量、错误率、延迟、饱和度。
2. 入口：API、页面、任务的核心体验指标。
3. 依赖：数据库、缓存、队列、第三方接口。
4. 版本：最近部署、实例数量、错误是否集中在新版本。

当告警触发时，值班人员应该能从总览进入具体依赖，再跳转到日志和 Trace。

## 最小闭环

告警应该链接到对应仪表板、日志查询和处理手册，让值班人员能够快速进入定位流程。

一个完整闭环通常包括：

- 指标发现异常。
- 告警通知负责人。
- 仪表板确认影响范围。
- Trace 定位慢点或失败调用。
- 日志查看具体错误上下文。
- Runbook 指导缓解动作。
- 事后复盘沉淀新的监控或自动化修复。

## 结语

可观测性是一套工程能力，不是一组工具清单。日志、指标和链路追踪只有串起来，才能帮助团队在复杂系统里快速定位问题。真正成熟的后端系统，应该让故障从“凭经验猜”变成“沿证据查”。
