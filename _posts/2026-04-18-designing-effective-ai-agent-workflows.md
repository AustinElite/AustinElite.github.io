优秀的 Agent 工作流不是让模型无限思考，而是在每个阶段提供清晰的输入、输出和停止条件。Agent 的价值在于把复杂任务拆成可执行步骤，但如果缺少边界，它很容易变成不可预测的循环：反复规划、反复调用工具、反复消耗 Token，却没有可靠结果。

一个有效的工作流应该像后端服务一样设计：输入明确、状态可见、失败可分类、结果可验证。

## Plan

规划阶段只负责拆分目标和识别依赖，不执行外部副作用。

规划结果最好是结构化的，例如：

```json
{
  "goal": "分析最近一周的错误日志",
  "steps": [
    {"type": "query_logs", "reason": "获取异常样本"},
    {"type": "summarize", "reason": "归类根因"},
    {"type": "draft_report", "reason": "生成处理建议"}
  ],
  "requiresApproval": false
}
```

这样后端可以在执行前检查步骤数量、工具权限和预算。规划不是越长越好，计划应当足够具体，但不能把业务判断完全交给模型。

## Act

执行阶段调用白名单工具，并在服务端校验参数。工具应该是窄接口，而不是“万能执行器”。例如不要给 Agent 一个可以执行任意 SQL 的工具，而是提供 `queryOrderStatus`、`searchErrorLogs`、`createDraftReply` 这样的限定能力。

每个工具都应该定义：

- 输入 Schema。
- 是否只读。
- 最大执行时间。
- 是否需要人工确认。
- 错误类型和可重试策略。

工具返回值也要尽量结构化。模型不需要拿到完整日志，只需要拿到可用于下一步推理的摘要、计数、样本和链接。

## Observe

观察阶段将工具结果转换成模型可理解的结构，同时保留原始数据用于审计。

例如一次日志查询可以返回：

- 总错误数。
- Top 5 异常类型。
- 每类异常的示例 Trace ID。
- 时间分布。
- 相关服务和版本。

这样模型能继续分析，而工程团队也能回到原始 Trace 里验证结论。不要只保存模型的自然语言总结，否则后续很难判断它是否遗漏了关键证据。

## Reflect

重试必须设置次数、预算和失败分类。没有边界的反思循环只会浪费 Token。

建议把失败分成三类：

1. **模型判断失败**：计划不合理、输出格式错误、遗漏约束。
2. **工具执行失败**：网络超时、第三方 API 错误、权限不足。
3. **业务规则失败**：目标不允许执行、状态不满足、需要人工确认。

不同失败类型对应不同处理方式。模型判断失败可以重新规划，工具失败可以指数退避重试，业务规则失败则应该停止并给出明确说明。

## Memory

记忆不是把所有历史对话都塞进上下文。更稳妥的方式是把记忆拆成几类：

- 用户偏好，例如语言、格式、常用仓库。
- 项目事实，例如服务名称、数据库表、部署环境。
- 任务历史，例如某次分析的结论和处理结果。

短期任务上下文可以随任务结束而清理，长期记忆必须支持查看、更新和删除。尤其在涉及隐私或业务数据时，记忆层需要比 Prompt 更严格的权限控制。

## Human-in-the-loop

并不是所有自动化都要全自动。对于高风险动作，可以把 Agent 设计成“草稿生成器”或“建议系统”：

- 自动分析问题，但人工确认修复命令。
- 自动生成邮件，但人工点击发送。
- 自动整理 PR 评论，但人工决定是否修改。

这种设计并不会削弱 Agent，反而能让它更快进入真实工作流。可靠系统的目标不是炫技，而是让用户敢用。

## Evaluation

Agent 工作流上线后，需要持续评估。可以收集成功率、平均步骤数、工具失败率、人工接管率和用户反馈。每次失败都应该能回放完整轨迹：模型看到了什么、计划了什么、调用了什么、为什么失败。

只有能被评估的 Agent，才有可能被持续优化。

## 结语

有效的 Agent 工作流不是“让模型自己想办法”，而是把模型放进一个可控的执行框架里。规划、执行、观察、反思、记忆和人工确认共同组成闭环。这个闭环越清晰，Agent 就越可靠，也越容易在真实业务中产生价值。
